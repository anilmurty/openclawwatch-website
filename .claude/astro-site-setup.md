# TokenJam Website — Astro Blog/Docs Setup Spec

**Repo:** github.com/Metabuilder-Labs/tokenjam-website
**Live site:** tokenjam.dev
**Current state (verified):** single-file static landing page (`index.html`, ~2000 lines, all CSS/JS inline), one legacy Vercel serverless function (`api/waitlist.js`, Resend audience signup — no UI currently calls it; kept for possible future reuse), Vercel-hosted with `cleanUrls: true`. No build step today.

## Goal

Add an Astro-based **blog** (and a minimal docs placeholder) to this repo **without touching the existing landing page or the legacy `api/waitlist.js` function**. After this task:

- `/` keeps serving the current `index.html` byte-for-byte.
- `/api/waitlist` still resolves (function preserved for future reuse; no UI uses it today).
- `/blog`, `/blog/[slug]`, `/blog/[slug].md`, `/blog/rss.xml`, `/docs`, `/llms.txt`, `/llms-full.txt`, `/sitemap-index.xml`, `/robots.txt` all work, generated by Astro at build time.

Out: writing actual posts, dynamic OG images, newsletter integration, docs content beyond a placeholder, scheduled-publishing cron. See "Out of scope" at the bottom.

## Why Astro (not Next.js + Fumadocs)

We rejected a full Next.js migration because (a) we have no docs content yet, (b) the landing page works and shouldn't be touched, (c) Astro's static output is faster and AI-crawler-friendly, (d) Astro coexists with the existing `index.html` and `api/` without rewrites. If TokenJam grows real docs + in-docs AI later, we can migrate to Fumadocs while keeping URLs stable.

## Phase 1 — Astro scaffold

### 1.1 Scaffold via temp directory (do not run `pnpm create astro` directly in repo root)

The official installer is unreliable inside a non-empty repo. Instead:

1. `pnpm create astro@latest /tmp/tj-astro -- --template minimal --typescript strict --install --no-git`
2. Copy these into the repo root: `astro.config.mjs`, `tsconfig.json`, `src/` (empty initial layout fine), and merge `package.json` (`dependencies`, `devDependencies`, `scripts`) into a new `package.json` at repo root.
3. `pnpm install` at repo root.
4. Verify `pnpm dev` starts and serves a hello-world Astro page.

### 1.2 Move static assets into `public/`

```
mkdir -p public
git mv index.html public/index.html
git mv favicon.svg favicon-32.png apple-touch-icon.png og-image.png github-social.png icon.svg public/
git rm robots.txt sitemap.xml      # they will be regenerated by Astro
```

Do **not** touch `api/`. Vercel routes `api/*` independently of Astro's `dist/` output; they coexist.

After the move:

- `pnpm dev` and confirm `/` serves the existing landing page identically (Astro falls through to `public/index.html` because there is no `src/pages/index.astro`).
- Hard-refresh and verify hero animation, terminal mock, and theme toggle all still work locally.

### 1.3 Astro config

`astro.config.mjs`:

```js
import { defineConfig } from 'astro/config';
import mdx from '@astrojs/mdx';
import sitemap from '@astrojs/sitemap';

export default defineConfig({
  site: 'https://tokenjam.dev',
  output: 'static',
  build: { format: 'directory' },     // matches Vercel cleanUrls
  trailingSlash: 'never',           // matches Vercel cleanUrls: no-slash canonical; fallback if 'ignore' misbehaves
  integrations: [
    mdx(),
    sitemap({ filter: (page) => !page.includes('/draft/') }),
  ],
  markdown: {
    shikiConfig: { theme: 'github-dark', wrap: true },
  },
});
```

```
pnpm add @astrojs/mdx @astrojs/sitemap @astrojs/rss
```

**No Tailwind.** The existing site is plain CSS with a well-defined token system; the blog will reuse those tokens via a single `src/styles/global.css`. Prose styles for MDX content go in the same file under a `.prose` class — small, scoped, no framework needed.

**URL canonicalization gotcha.** `build.format: 'directory'` writes `/blog/foo/index.html`. Vercel `cleanUrls: true` serves that at `/blog/foo` and 308-redirects `/blog/foo/` → `/blog/foo`. `trailingSlash: 'never'` makes Astro emit canonicals and internal links without slashes so everything agrees. If after deploy you find that `/blog/foo/` serves a 200 with duplicate content (instead of redirecting), that's the duplicate-content trap — fix in `vercel.json` with an explicit redirect before search engines index both forms.

**Token duplication is accepted for v1.** Color/font tokens live both inline in `public/index.html` and in `src/styles/global.css`. The list is small and stable; manual sync on change is fine. Note this explicitly in CLAUDE.md so future edits remember to update both. Extracting to a shared CSS file is rejected for v1 — the risk to the landing page outweighs the maintenance saving.

## Phase 2 — Content collections

### 2.1 `src/content/config.ts`

```ts
import { defineCollection, z } from 'astro:content';

const blog = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    description: z.string(),
    author: z.string().default('anil'),
    publishDate: z.coerce.date(),
    lastUpdated: z.coerce.date().optional(),
    tags: z.array(z.string()).default([]),
    pillar: z.enum([
      'foundational','observability','evaluation','environments',
      'gateways','memory','guardrails','hitl','control-plane',
      'optimization','comparison','thesis',
    ]).optional(),
    image: z.string().optional(),
    draft: z.boolean().default(false),
    canonicalUrl: z.string().optional(),
  }),
});

const docs = defineCollection({
  type: 'content',
  schema: z.object({
    title: z.string(),
    description: z.string(),
    order: z.number().default(0),
    section: z.string().optional(),
    lastUpdated: z.coerce.date().optional(),
  }),
});

export const collections = { blog, docs };
```

Content lives at `src/content/blog/*.mdx` and `src/content/docs/*.mdx` (Astro's default; not at top-level `content/`).

### 2.2 Helpers

`src/lib/blog.ts`:

```ts
import { getCollection } from 'astro:content';

export async function getPublishedPosts() {
  const now = new Date();
  const posts = await getCollection('blog', ({ data }) => !data.draft && data.publishDate <= now);
  return posts.sort((a, b) => +b.data.publishDate - +a.data.publishDate);
}
export async function getPostsByTag(tag: string) {
  return (await getPublishedPosts()).filter((p) => p.data.tags.includes(tag));
}
export async function getPostsByAuthor(slug: string) {
  return (await getPublishedPosts()).filter((p) => p.data.author === slug);
}
```

The `publishDate <= now` filter at build time is what makes scheduled publishing work: future-dated posts are excluded until the next build after their date.

## Phase 3 — Layouts and pages

### 3.1 `src/layouts/BaseLayout.astro`

Provides:

- `<!doctype html>`, `<html lang="en">`.
- `<head>`: charset, viewport, theme-color, canonical, default OG + Twitter tags (overridable per page), favicon links pointing at the files now in `public/`, Geist + Geist Mono preload, GA snippet (`G-C3WCWQW0WR`), and the **theme-init `<script>` copied verbatim from `index.html`** (the one that reads `tj-theme` and sets `data-theme-pref`/`data-theme` before paint). It must be an **inline `<script>` with no `src=`, no `defer`, no `async`, no `type="module"`** — synchronous, blocking, before `<body>`. Anything else causes FOUC on hard reload.
- Imports `src/styles/global.css` (the design tokens).
- Renders `<Nav />`, `<slot />`, `<Footer />`.

### 3.2 `Nav.astro` and `Footer.astro`

Match the existing landing-page nav/footer markup and styles so blog/docs feel like the same site. Nav links: `/`, `/blog`, GitHub. (No `/docs` link in the nav until docs are real — the route exists but is intentionally low-profile.) Reuse the existing theme-toggle component markup.

### 3.3 Blog index — `src/pages/blog/index.astro`

- Calls `getPublishedPosts()`.
- Renders a list of cards: title, description, date, author, pillar, reading-time estimate.
- Top-of-page tag/pillar filter (client-side only).
- RSS `<link rel="alternate">` in head.
- v1: just show all (or last 30). Add real pagination only if/when post count justifies it.

### 3.4 Blog post — `src/pages/blog/[slug].astro`

Standard `getStaticPaths` over published posts. Wraps in `BlogLayout.astro` which renders:

- H1 + byline (author link → `/blog/authors/[author]`, publishDate, lastUpdated, reading time).
- `<CopyForAI />` button (see 4.7).
- `<Content />` with `.prose` styling.
- Tags + 3–5 related posts by tag overlap.
- `<Schema />` JSON-LD (BlogPosting; see 4.2).
- OG image: `post.data.image` if set, else `/og-image.png`.
- Canonical: `post.data.canonicalUrl` if set, else `Astro.url`.

### 3.5 Author and tag pages

`src/pages/blog/authors/[author].astro` and `src/pages/blog/tags/[tag].astro` — both static, enumerated via `getStaticPaths`.

### 3.6 RSS — `src/pages/blog/rss.xml.ts`

Standard `@astrojs/rss` usage over `getPublishedPosts()`, full body in `content` field.

### 3.7 Docs placeholder

Single file `src/content/docs/index.mdx`:


## Phase 4 — AEO infrastructure

### 4.1 MDX components (imported per-post; no global registration)

In `src/components/`:

- `TLDR.astro` — top-of-post summary block.
- `DefinitionBox.astro` — `term` prop, optional `DefinedTerm` JSON-LD.
- `ComparisonTable.astro` — `tools` + `rows` props, semantic `<table>`.
- `FAQBlock.astro` — `items: {question, answer}[]`, renders as `<dl>` and emits FAQPage JSON-LD.
- `Callout.astro` — `type: 'info' | 'warn' | 'tip'`.
- `CopyForAI.astro` — see 4.7.
- `Schema.astro` — renders `<script type="application/ld+json">`.

Note: Astro does not auto-inject MDX components. Posts that use these import them at the top of the `.mdx` file. That's fine.

### 4.2 Schema helpers — `src/lib/schema.ts`

`articleSchema(post)`, `organizationSchema()`, `softwareApplicationSchema()`, `breadcrumbSchema(items)`, `faqPageSchema(items)`, `personSchema(author)`. All return plain JS objects; `<Schema>` serializes.

Apply: BlogPosting on each post; Person on author pages; BreadcrumbList on docs; Organization on blog index. Landing page already has its own; don't duplicate there.

### 4.3 Sitemap

`@astrojs/sitemap` produces `dist/sitemap-index.xml` automatically. Because we deleted the old root `sitemap.xml`, no shadow. If external tools expect `/sitemap.xml`, add a Vercel rewrite (see Phase 5).

### 4.4 Robots — `src/pages/robots.txt.ts`

Generated route returning `text/plain` allow-list for: `*`, `GPTBot`, `ClaudeBot`, `Claude-Web`, `anthropic-ai`, `PerplexityBot`, `GoogleOther`, `Google-Extended`, `Bingbot`, `CCBot`, `cohere-ai`, `meta-externalagent`, `Applebot-Extended`. Footer: `Sitemap: https://tokenjam.dev/sitemap-index.xml`.

### 4.5 `llms.txt` and `llms-full.txt`

`src/pages/llms.txt.ts` — short index: TokenJam summary, then per-post and per-doc lines pointing at the `.md` URLs.
`src/pages/llms-full.txt.ts` — concatenated raw markdown bodies of all published posts and docs.

### 4.6 `.md` endpoints

`src/pages/blog/[slug].md.ts` and `src/pages/docs/[...slug].md.ts` — `getStaticPaths` over the same set as the HTML pages, return `post.body` with `Content-Type: text/markdown`. JSX components in `.mdx` will appear as raw JSX text in the `.md` output; that's acceptable for AI consumers in v1. AST cleanup is deferred.

### 4.7 Copy for AI

`CopyForAI.astro` — button at top of post/doc pages. Click handler fetches the current path with `.md` appended and writes to clipboard. Show a brief toast on success/failure.

### 4.8 Dynamic OG images — **deferred**

Static `output` cannot run `@vercel/og` at request time. Either move to `output: 'hybrid'` + a serverless function (extra complexity) or skip for v1. **v1: use `/og-image.png` as the static fallback for all blog posts.** Revisit if/when post-level OG becomes worth the deploy-config change.

### 5.5 CLAUDE.md update

Replace contents with a description of the **hybrid** layout: `public/index.html` is the landing page (do not modify), `api/waitlist.js` is the Resend serverless function (env vars `RESEND_API_KEY`, `RESEND_AUDIENCE_ID`), Astro handles `/blog/*` and `/docs/*` from `src/content/`. Document: how to add a post (filename `YYYY-MM-DD-slug.mdx`, frontmatter shape, `publishDate` controls scheduled publishing requiring a rebuild), where AEO files live, and the design system (Geist/Geist Mono, the `--navy/--blue/...` token set, theme key `tj-theme`).

### 5.6 Scheduled publishing — **deferred**

Future-dated posts only appear after a rebuild. Defer setting up a Vercel cron until there's at least one scheduled post. When ready: add `crons` to `vercel.json` hitting a new `api/rebuild.js` that calls a Vercel deploy hook (env: `CRON_SECRET`, `VERCEL_DEPLOY_HOOK_URL`). Until then, redeploy manually.
